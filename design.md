# DESIGN.md — Controlled AI Software Development Pipeline

> Это запись проектных решений (decision record) универсального фреймворка.
> Источник истины по архитектуре. Любая новая рабочая сессия (opencode / Claude Code /
> новый чат) должна стартовать с прочтения этого файла.
> Статус: дизайн зафиксирован, реализация ещё не начата.

---

## 1. Цель

Не multi-agent framework, а **controlled AI software development pipeline**:
минимальный входной промпт → расширение требований → multi-step разработка →
независимая проверка → готово. Линейный pipeline + строгие роли + цикл проверки.

Принципы:
- Double-check вместо консенсуса (независимая проверка, а не «много агентов согласились»).
- Контекст через файлы, не через chat memory (LLM stateless, состояние в артефактах).
- Жёсткие роли: каждый агент делает одну вещь, роли не пересекаются.
- Всегда есть stop-condition (иначе бесконечные «улучшения»).
- Memory = артефакты проекта, а не «память в голове модели».

---

## 2. Выбор инструмента

**Решение: opencode.**

Причина: нативно даёт ровно нужные примитивы — primary/subagent, модель на каждого
агента (любой провайдер, формат `provider/model-id`), tool-permissions на каждого агента
(read-only роли, bash где нужно), markdown-определения агентов, AGENTS.md для проектного
контекста. Терминальный, 4-ролевой pipeline ложится без внешнего оркестратора.

Рассмотренные альтернативы и почему не сейчас:
- **Kilo Code CLI** — Roo-style orchestrator-режим в терминале, форк Roo, CLI поверх
  opencode. Реальный апгрейд, если ручная склейка ролей в opencode заболит. Держать в уме.
- **Forge** — встроенный трёхролевой split (implement / research read-only / plan), BYOK
  300+ моделей. Концептуально близко, но новый Rust-тул.
- **Roo Code** — самая зрелая оркестрация (file-regex permissions, Boomerang), но это
  плагин VS Code, уводит из терминала.
- **Aider** — architect/editor split, бенчмарк-проверенный (DeepSeek силён как Editor —
  независимое подтверждение нашей раскладки), но это 2-ролевая пара, не 4-ролевой
  reviewer-гейт.
- **Claude Code / Codex / Qwen / Antigravity** — заперты в своём провайдере, не дают
  mix Claude+DeepSeek. Мимо.

**Когда пересмотреть:** если ручная оркестрация петли (кто кого зовёт, передача контекста
между ролями) превращается в боль — смотреть Kilo Code CLI как апгрейд по прямой
(навык opencode не пропадает, Kilo построен поверх него).

## 3. Роли и распределение моделей

Четыре роли. Принцип распределения: независимость пары Implementer↔Reviewer
(обязательно разные семейства, иначе double-check фиктивен) + экономия на самой
токеноёмкой роли.

| Роль | Модель | Почему |
|------|--------|--------|
| Intake / Requirement Expander | Claude | Запуск раз на фичу, объём мизерный; работа тонкая (размытое → точное, вытащить допущения) |
| Planner | Claude | Высоколеверджевая: плохой план травит весь поток вниз, ошибку трудно поймать |
| Implementer | DeepSeek | ~80% всех токенов (пишет/переписывает код), дешёвая массовка |
| Reviewer | Claude | Гейт качества, потолок системы; обязательно другое семейство, чем Implementer |

Claude стоит на трёх ролях, но две из них (Intake, Planner) почти бесплатны из-за низкого
объёма. Дорогой токен-сток (Implementer) — DeepSeek. Итог: claude-уровень качества при
deepseek-цене на основной массе.

Описание ролей:
1. **Intake / Expander** — превращает ленивый промпт в spec, фиксирует assumptions.
   Output: `requirements.md`. (Уточняющие вопросы — НЕ интерактивно, см. §5.)
2. **Planner** — архитектура + декомпозиция на задачи + acceptance-тесты на задачу.
   НЕ пишет код. Output: `plan.md`, `architecture.md`, тесты.
3. **Implementer** — пишет код по `plan.md`. НЕ меняет архитектуру.
4. **Reviewer** — независимая проверка того, что тесты не ловят. APPROVE / REJECT(+причина).

## 4. Поток и два вложенных цикла

```text
Intake (Claude) ──> requirements.md ──[human gate]──>
Planner (Claude) ──> plan.md + acceptance-тесты   (тесты авторства Claude, НЕ DeepSeek)
        │
        ▼  по задачам из plan.md (гранулярность = задача, не весь кодбейз)
  ┌──────────────────────────────────────────────┐
  │ ВНУТРЕННИЙ ДЕШЁВЫЙ ЦИКЛ                        │
  │ Implementer (DeepSeek):                        │
  │   пишет код → сам гоняет тесты/линтер (bash)   │
  │   красные? чинит сам, Claude НЕ зовётся        │
  └──────────────────────────────────────────────┘
        │ только когда тесты зелёные
        ▼
  ┌──────────────────────────────────────────────┐
  │ ВНЕШНИЙ ДОРОГОЙ ЦИКЛ                           │
  │ Reviewer (Claude): судит то, что тесты не ловят│
  │   edge cases, соответствие плану, дизайн       │
  │   может флагать ДЕФЕКТ ПЛАНА (→ назад Planner)  │
  │   APPROVE / REJECT(+причина)                   │
  └──────────────────────────────────────────────┘
        │ APPROVE
        ▼  следующая задача
```

Ключевые механики:
- **Тесты пишет Planner (Claude), не Implementer** — иначе DeepSeek протестит ровно то,
  что и так работает (correlated oracle). Оракул должен быть авторства другой стороны.
- **Reviewer видит только код, прошедший тесты** — его дорогие токены тратятся на
  субъективное суждение, а не на ловлю опечаток.
- **Plan-defect flagging** — Reviewer вправе флагать дефект *плана*, а не только кода →
  эскалация к Planner. Лечит слабую корреляцию Claude-planner ↔ Claude-reviewer.
- **Обратное ребро Implementer → Planner** — если план нереализуем как написан,
  Implementer поднимает «plan defect», а не молча меняет архитектуру.

## 5. Контроль цикла и стоп-условия

- Лимит **2–3 итерации**, потом решение человека.
- Эскалация выносит человеку **конкретное разногласие**, а не «не получилось».
- Различать два типа незакрытого ревью:
  - *один и тот же баг повторяется* → Implementer не может починить → эскалация к человеку.
  - *каждый раз новый баг* → петля не сходится → задача плохо декомпозирована → назад к Planner.
- Гранулярность ревью = **одна задача** (привязка к декомпозиции в `plan.md`).
  Стоп-условие появляется само: задача закрыта → следующая.
- **Intake-чекпойнт человека — на гейте `requirements.md`, не интерактивным Q&A.**
  Expander сам делает documented assumptions; человек читает/правит requirements.md;
  только потом стартует Planner. Так «минимальный промпт» сохраняется, контрольная точка есть.

## 6. Три слоя контекста (критично против prompt coupling)

Разделение, нарушение которого делает систему хрупкой:

- **Промпт роли** — *как ведёт себя роль*, generic, проектонезависимо. Живёт в определении
  агента (`.opencode/agents/<role>.md`). Проектных фактов тут НЕТ.
- **AGENTS.md** — *стабильные факты про этот проект* (стек, команды build/test/lint,
  конвенции, неочевидные паттерны). Читается всеми ролями автоматически. Поведения ролей
  тут НЕТ — иначе каждая роль засоряется инструкциями всех остальных.
- **workspace/*.md** — *меняющиеся факты про текущую фичу* (requirements, plan,
  architecture, decisions). Динамическая память.

Правило-разделитель: поведение → в агента; стабильное про проект → в AGENTS.md;
эволюционирующее про задачу → в workspace.

## 7. Структура workspace

```text
workspace/
  requirements.md    # output Intake, гейтится человеком
  plan.md            # output Planner: декомпозиция на задачи
  architecture.md    # архитектура / текущее состояние кодбейза
  decisions.md       # append-only, с обоснованием (анти-drift)
  src/
  tests/             # acceptance-тесты авторства Planner
```

---

## 8. Permissions (per-agent в opencode)

- **Planner** — read-only на запись кода (может писать только в plan/architecture/тесты).
- **Reviewer** — read-only на код + **bash** (гоняет тесты/линтер).
- **Implementer** — write/edit код + **bash** (гоняет тесты в внутреннем цикле).
- **Intake** — пишет только requirements.md.

## 9. Проектная подстройка

Обязательный добавочный слой — **AGENTS.md** (тот самый «как работает этот проект»,
которого не было в исходной схеме). Отсюда Reviewer/Implementer берут точную команду
прогона тестов.

- opencode читает AGENTS.md автоматически, поднимаясь вверх от текущей директории;
  иерархичный (ближайший к коду файл выигрывает). Создаётся командой `/init`.
- **Держать коротким**: root AGENTS.md < 200 строк (на практике 30–50).
- **НЕ доверять авто-генерации.** Исследование ETH: LLM-сгенерированные контекст-файлы в
  5 из 8 сетапов *ухудшали* результат (+2.45–3.92 шага, +20–23% стоимости). Написанные
  человеком — помогают. Вывод: `/init` как черновик, дальше вычитывать и резать руками.

Механизмы поменьше (лёгкие, но без них brownfield и повторные запуски ломаются):
- **Onboarding (brownfield)** — разовый read-only проход: заполняет `architecture.md`
  базовым состоянием + черновик AGENTS.md. Greenfield пропускает.
- **Lifecycle / идемпотентность** — если артефакты уже есть, роли *читают и дополняют*,
  не пересоздают. Обновление AGENTS.md коммитится вместе с кодом (анти-drift).
- **Тонкий per-project knob** — `.opencode/` проекта + опциональные model-overrides на роль.
  Только команды + модели, без DSL. Держать минимальным (риск overengineering).

## 10. Двух-репная архитектура и генерация

Цель (пункт назначения, НЕ первый шаг):
- **Репо A — универсальный фреймворк**: человекописаные шаблоны агентов + скелет AGENTS.md.
- **Репо B — инстанс в проекте**: сгенерированный `.opencode/agents/` + AGENTS.md + workspace.

Генерация = **тонкий скаффолдер** (cookiecutter-стиль): копирует шаблоны, подставляет
факты (стек, команды test/lint, какие роли активны). НЕ LLM-генератор промптов
(дорого, drift инстансов от шаблона, импорт деградации из исследования ETH, overengineering).

Проектные *поведенческие* отклонения (напр. security-critical Reviewer с другим чек-листом)
— через ручной per-project override: свой `reviewer.md` в `.opencode/agents/` проекта
перекрывает глобальный. Генератор для этого не нужен.

---

## 11. Последовательность реализации (важнее самой архитектуры)

**Генератор сейчас строить нельзя — нечего шаблонизировать.** Точки вариативности видны
только после живого прогона. Правильный порядок:

1. Написать конкретные **generic-промпты ролей** как обычные файлы.
2. Обкатать на **одном реальном проекте**.
3. Посмотреть, что *на самом деле* меняется от проекта к проекту.
4. Только потом извлечь шаблоны + написать скаффолдер.

Шаблон — дистилляция уже работающего, а не отправная точка.
Генератор — фаза 2, после первого живого прогона.

---

## 12. Отслеживаемые риски

- correlated errors (двойная проверка ≠ гарантия независимости) → лечим разными семействами + plan-defect flag.
- drift целей в циклах review/fix → лечим per-task гранулярностью + различением типов реджекта.
- overengineering requirements / инструмента → лечим «prompts first, generator later» + минимальным knob.
- prompt coupling / brittle system → лечим тремя слоями контекста (§6).
- loop без stop-condition → лимит итераций + эскалация человеку.
- memory / context drift (устаревшие AGENTS.md, workspace) → lifecycle-правило + append-only decisions + коммит AGENTS.md с кодом.

## 13. Открытый вопрос / следующий шаг

Написать первый реальный промпт роли. **Развилка: Reviewer или Intake?**
- *Reviewer* — самый тонкий, определяет потолок качества (формат REJECT, граница
  «что ловят тесты vs что судит модель», plan-defect flag). Аргумент: задаёт планку,
  под которую подстраиваются остальные.
- *Intake* — начало потока, проще, даёт быстрый первый прогон end-to-end.
